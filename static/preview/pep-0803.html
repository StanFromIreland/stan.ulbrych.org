PEP 803 – Creation of a CPython mirror

Abstract

This PEP proposes the creation of an official read-only mirror of the CPython Git repository, currently hosted on GitHub (since PEP 512), on another hosting platform.

Rationale

The authors believe that mirroring the repository on an alternative platform, will improve the security, resilience, and accessibility of CPython without impacting the current development workflow. Currently, the up-to-date CPython source code is officially available only on GitHub, following the retirement of hg.python.org.

Many open source projects provide mirrors, for example Linux and FreeBSD.

Note

The standard clone of a repository does not pull all information, see the Git documenation for more information.

Depending on a single storage provider carries the risk, that if the provider were to go down, the data stored on the platform is lost. The same reasoing can be used for backing up a personal computer, Photos, documents and so on. Even during the case of a temporary outage, development can still continue as one can pull the most recent state of the repository from the mirror.

Some people prefer using open source projects for the development of open source projects, GitHub is not an open source project, a mirror will allow for us to change that.

Different hosting providers may have less restrictions, may it be govermental, corporate or otherwise, it would make the source more accesiable.

The mirror is not intended to replace the host for contributions (issues, pull requests, CI), but to provide an additional read-only distribution channel for the Git data (commits, branches, tags etc.) as having multiple contribution platforms would introduce far too much complexity and asynchrony.

In the case that GitHub were to go down permanetly, having a mirror with the python namespace would allow for a much smoother migration to a new contribtuion platform.

Mirrors are generally low maintaince, and should not have a financial impact on the PSF, as there are many free hosting providers.

Specification

The repository will be mirrored using Git’s –mirror option, either using a platforms builtin integration or a script.

The fact that the repository is a mirror will be clearly noted in the description as so to avoid possible confusion.

Security Implications

This PEP increases the security workload, as two platforms must be secured. The mirror being read-only alleviates some risk of compromization, but not all.

Open Questions

What platform should be used? Ideally free, as so to not have a financial impact on the PSF.
